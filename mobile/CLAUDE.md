# Mobile (Flutter) — Agent Conventions

This file is read by Claude Code when working inside `mobile/`. The root
`CLAUDE.md` covers cross-stack rules (Firebase auth, design tokens, deploy);
this one covers Flutter-specific layout and patterns. **Both apply.**

## Folder layout

Strict **feature-first** layout. Don't add files to `lib/` directly except
`main.dart`, `app.dart`, `firebase_options.dart`. Everything else lives under
`core/` (cross-feature) or `features/<name>/` (feature-scoped).

```
lib/
├── main.dart                    # bootstrap: Firebase init + native splash + runApp
├── app.dart                     # MaterialApp.router; reads routerProvider
├── firebase_options.dart        # generated by `flutterfire configure` — don't edit
├── core/
│   ├── design/tokens.dart       # generated by design/build.mjs — don't edit
│   ├── domain/                  # cross-feature models (AppUser, Role)
│   ├── log.dart                 # logd() — debug-only logger
│   ├── network/                 # ApiClient, NetworkException, apiClientProvider
│   ├── router/app_router.dart   # go_router config + auth-driven redirect
│   └── widgets/                 # cross-feature widgets (buttons, dialogs, layouts)
└── features/
    └── <name>/
        ├── data/                # *_api.dart (instance class) + *_api_provider.dart
        ├── domain/              # feature models, validators, value objects
        └── presentation/
            ├── controller/      # Riverpod controllers (AsyncNotifier / Notifier)
            ├── pages/           # ConsumerStatefulWidget screens
            └── widgets/         # feature-specific widgets
```

**Boundaries:**
- `features/<x>/` may import from `core/` and from another feature's
  `domain/` (data classes are safe to share). Don't import another feature's
  `data/` or `presentation/` — that's a circular-dependency trap.
- `core/` may not import from `features/`. Ever. If a "core" thing depends on
  a feature, it isn't core.

## Adding a new feature

For a new feature `foo`:

1. Create `features/foo/{data,domain,presentation/{controller,pages,widgets}}`.
2. **Data layer** — instance class wrapping `ApiClient`:
   ```dart
   // features/foo/data/foo_api.dart
   class FooApi {
     final ApiClient _client;
     FooApi(this._client);

     Future<List<Foo>> list(String idToken) async {
       final res = await _client.get('/foo', idToken: idToken);
       if (res.statusCode != 200) throw HttpException('GET /foo: ${res.statusCode}');
       // parse and return
     }
   }
   ```
   Plus a provider:
   ```dart
   // features/foo/data/foo_api_provider.dart
   final fooApiProvider = Provider<FooApi>((ref) => FooApi(ref.watch(apiClientProvider)));
   ```
3. **Controller** — `AsyncNotifier` (state) or plain methods on a `Notifier`,
   reading the API via `ref.read(fooApiProvider)`. Mirror `AuthController` —
   it's the canonical example.
4. **Page** — `ConsumerStatefulWidget`. Imports go: stdlib → packages →
   `core/` → same feature.
5. **Route** — add a `GoRoute` to `core/router/app_router.dart` and update
   the redirect logic if the route should be auth-gated.

## Adding a new API endpoint to an existing feature

- Add a method on the feature's `*Api` class. Use `_client.get` /
  `_client.postJson` / `_client.sendMultipart`.
- Status-code interpretation (e.g. "404 = not found" vs "404 = error") is the
  Api class's job. The `ApiClient` only translates network-level failures to
  `NetworkException`.
- Backend error codes (`{"error": "manager_code_not_found"}`) → map them to
  typed exceptions per feature (see `RegistrationException`,
  `PasswordResetException` in `features/auth/data/auth_api.dart`).

## Adding a new widget

**Before creating a new widget, search the existing widget folders first.**
- `core/widgets/` → cross-feature (buttons, dialogs, layout primitives)
- `features/<x>/presentation/widgets/` → feature-scoped (form fields, etc.)

Promote a widget from `features/x/` to `core/widgets/` only when a *second*
feature wants to import it. Don't promote speculatively.

## Routing — go_router

- Single source of truth: `core/router/app_router.dart`.
- Navigate via `context.go('/path')` (replace stack), `context.push('/path')`
  (push on top), or `context.pushReplacement('/path')` (replace top).
- Pass non-trivial params via `extra:` (e.g. `PendingGoogleProfile`,
  `resetToken`). Avoid query params for sensitive data (PII like email).
- Auth gating happens in the redirect block — do NOT add manual auth checks
  in pages. If your page should not be reachable while logged out, add it to
  the redirect logic.
- Don't call `Navigator.push(MaterialPageRoute(...))` anywhere in feature
  code — it bypasses the router and produces inconsistent back-stack
  behavior. Bottom sheets and dialogs (`showDialog`, `showModalBottomSheet`)
  remain on the standard `Navigator` because they're not routes.

## State management — Riverpod

- Controllers are `AsyncNotifier<T>` (when state is async/loadable) or
  `Notifier<T>` (when sync). See `AuthController` for the pattern.
- Read providers in widgets via `ref.watch` (rebuild on change) or
  `ref.read` (one-shot, e.g. button handlers calling a method).
- Mutate state inside the Notifier with `state = AsyncData(...)` or
  `state = ...`. Never expose mutable state from the controller's public API.
- For testing: providers can be overridden via `ProviderScope(overrides: [...])`.
  All `*Api` classes are instance classes precisely so they can be swapped.

## Network errors

Always convert to a typed exception at the boundary:
- Transport failure (no connection, timeout) → `NetworkException`. Caught in
  pages to render "Нет соединения с сервером".
- Server-returned error codes → feature-specific exception class with the
  `code` string (e.g. `RegistrationException.code = 'manager_code_not_found'`).
  Pages switch on `code` to set the right field-level error.

## Logging

Use `logd(message, [error, stack])` from `core/log.dart`. It's a no-op in
release builds. Add it around:
- Network calls that swallow errors (best-effort cleanup, retry fallbacks).
- Auth-flow branches that aren't trivially debuggable from the UI.

Don't log passwords, tokens, or PII. Don't add a logging package — `debugPrint`
is enough until we have a real observability story.

## Internationalization

**Pre-launch decision (2026-05-01): Russian-only, no `flutter_intl` setup.**

- Hardcoded Russian strings are the convention. Don't introduce ARB files,
  `AppLocalizations`, or string-key indirection without explicit user buy-in.
- When a second language is requested, the migration path is `flutter_intl`
  + ARB files. It's deferred deliberately — adding it now without a user
  need is gold-plating.

## Design system

- **Never** hardcode colors. Use `AppColors.*` from `core/design/tokens.dart`.
- Tokens are codegen'd from `design/tokens.json` via `npm run design:build`
  (run from `web/frontend/`). Don't edit `tokens.dart` by hand.
- Spacing/radii/typography aren't yet tokenized — use raw values for now.
  Promote to tokens only when the same value appears in 3+ places.

## What NOT to do

- Don't add `jsonwebtoken`, `bcrypt`, or any auth library. Auth is Firebase.
- Don't bypass the router (no `Navigator.push(MaterialPageRoute(...))`).
- Don't make API calls from pages directly. Pages call controllers;
  controllers call `*Api` classes.
- Don't make `*Api` classes static. Instance + provider, always — for tests.
- Don't import `features/x/` from `core/`.
- Don't import another feature's `data/` or `presentation/` — only `domain/`.
- Don't add a state-management library other than Riverpod.
- Don't add a routing library other than go_router.
- Don't introduce a component kit (Material UI clones, etc.) — designs come
  from Figma; we build to match using `core/widgets/` + tokens.
- Don't add `flutter_dotenv`. Compile-time `--dart-define=API_URL=...` is the
  convention (see `ApiClient.resolveBaseUrl`).
- Don't unpin `google_sign_in_ios` (see root `CLAUDE.md` for the reason).

## Testing

No tests exist yet. When adding them:
- Unit-test controllers by overriding the feature's `*Api` provider with a
  fake. The instance-class design exists for this.
- Don't mock `package:http` directly — pass a fake `http.Client` into
  `ApiClient` via the constructor.

## Common pitfalls

- **`mounted` after async gap** — inside a closure (`onPopInvokedWithResult`,
  etc.), the analyzer may not connect `mounted` to `context`. Use
  `if (!context.mounted) return;` instead of `if (!mounted)`.
- **`context` after `await`** — always re-check `mounted` (or `context.mounted`)
  before using `context` after an await.
- **Firebase ID token freshness** — call `fbUser.getIdToken()` *per request*.
  It auto-refreshes near expiry. Don't cache the token.
- **Route extras** — go_router's `state.extra` is `Object?`. Cast carefully
  (`state.extra as String`, `state.extra as PendingGoogleProfile`). If the
  caller forgets `extra:`, you get a runtime cast error.
